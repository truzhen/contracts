# Truzhen 底层逻辑与哲学理念

> 本文件是 Truzhen 的理念总纲，用来解释系统为什么这样设计。它不是契约变更，不新增 schema，不改变 Go 类型，不声明下游已接线。涉及功能完成度、版本、发布和验收状态时，以六仓各自 `FEATURE_LEDGER.md`、治理文件和真实验证证据为准。

## 0. 本文件边界

本文件吸收三类输入：

1. 关于 AI Agent / 工具智能 / 主体性 / 可惩罚性的文章与讨论。
2. Owner 对 Truzhen 的定位：本地优先的人-事-判断事务操作层，AI 负责把判断前的一切准备好，人负责判断与抉择。
3. 六仓治理文件确认的边界：`truzhenos`、`truzhen-contracts`、`truzhen-packs`、`truzhen-software`、`truzhen-cloud`、`truzhen-client-web-desktop` 各自的真相源和禁止事项。

本文只讨论底层逻辑、产品哲学和架构原则，不替代任何仓库的实现治理。

## 1. 一句话定位

Truzhen 是本地优先的人-事-判断事务操作层。

它运行在 OS 之上、App 之上、协作软件之上、业务系统之间，把 AI、行业经验、外部软件和云服务编排成一条受主人主权约束的事务链：

```text
候选生成 -> 主权确认 -> 抉择推演 -> Base Gate 裁定 -> Gateway 受控执行 -> Receipt 回放 -> 记忆候选 -> Owner 确认后入 Memory
```

AI 助手的核心价值不是替人做判断，而是把判断之前的一切准备好：搜集、整理、记忆、推演、起草、列选项、列后果、生成候选和暴露风险。真正关键、只能由人做、也必须由人承担后果的动作，是判断与抉择。

## 2. Truzhen 的第一性问题

Truzhen 要回答的问题不是“怎样做一个更自主的 AI Agent”，而是：

```text
在 AI 不能担责、不能被真正惩罚、没有内生边界感的前提下，
怎样让 AI 仍然能安全地参与真实事务？
```

这个问题比模型能力更底层。

模型越强，只说明它能生成更复杂的候选、调用更多工具、覆盖更多场景。它不因此获得主体性，也不因此能承担责任。企业和小团队老板真正担心的不是 AI 不够聪明，而是出了事谁承担。只要责任不能转移给 AI，高权限业务就不能建立在“AI 会自己把握分寸”之上。

Truzhen 的答案是：不要把 AI 包装成主体。承认它是工具智能，然后在架构上把它接到人的主权链上。

## 3. 工具智能与主体智能的分野

文章里最值得吸收的不是“AI 永远不可能有主体性”这种强断言，而是它对当前 AI 工程边界的实用判断：

- AI 没有自己的存续利益。
- AI 不会因为损失、惩罚、羞耻、坐牢、破产或失去身份而改变行为。
- AI 不能真正承接责任。
- AI 的成本和风险都由使用它的人或组织承担。

因此，当前 AI 更像工具智能，而不是主体智能。

工具智能可以很强，可以写代码、做方案、起草文书、读资料、调用工具、生成流程。但工具智能不应被放到最终拍板的位置。它可以延伸人的能力，不能替代人的主权。

Truzhen 不需要证明“未来永远没有主体 AI”。Truzhen 只需要抓住当前产业落地最稳的事实：现在的 AI 不能担责，所以真实业务里的责任链必须落在人和组织身上。

## 4. 可惩罚性论证与 Truzhen 的工程回应

代理关系的底层不是“聪明”，而是“可追责”。人类代理人之所以能被委托，是因为他有自身利益和真实代价：可以被扣钱、撤职、起诉、禁业，严重时失去自由。代价落在他自己身上，所以他会在模糊地带权衡风险。

AI 没有这个锚点。删对话、停算力、销毁实例，都不会让 AI 产生真正的损失感。强化学习的数值惩罚也不是社会制度意义上的惩罚。AI 不能被真正惩罚，就不能真正承接责任。

Truzhen 对这条论证的回应不是写免责声明，而是把它做进架构：

| 可惩罚性论证指出的问题 | Truzhen 的工程回应 |
| --- | --- |
| AI 不可惩罚，责任无法转移 | AI 只生成 Candidate，Owner 才能做 Decision |
| AI 没有行为锚点，容易越过模糊边界 | Base Gate、SideEffectGate、SendGate、MemoryWriteGate 让越界在物理链路上失败 |
| AI 没有资源是自己的概念，容易无限推演 | 预算、轮次、任务边界、FutureShadow 和熔断由系统外部约束 |
| 出事后无法追责 AI | Receipt / Evidence 记录谁确认、什么候选、什么风险、什么结果 |
| AI 可能把工具调用包装成正式事实 | Candidate/Formal 隔离，正式事实只能由基座签发和回执证明 |

这里的 Decision 不等于每次都打断 Owner 让他当场点头。Owner 的决定可以是即时确认，也可以是 Owner 预先设定的标准策略，例如低风险联系人、低风险模板和授权范围内的自动放行；无外部副作用的本地留痕也可以只记录证据而不反复审批。后者本质上仍是 Owner 的决定，只是决定的时点提前到授权或策略设定时，而不是每次执行时。区别只在“何时拍板”，不在“谁拍板”。闸门永远只认 Owner 的决定，不认 AI 自己的判断。

所以 Truzhen 不是“更安全的自主 Agent”。Truzhen 从第一天就不做无人拍板的 Agent。它做的是带主权闸门、候选治理和回执追踪的工具智能事务层。

## 5. Truzhen 不是要和 OS 抢权限

Apple、Microsoft、Google 或其它 OS 第一方 AI 会天然强调：

- 更方便
- 更智能
- 更系统级
- 更无缝

Truzhen 不应该和 OS 比谁权限更大。Truzhen 要把“不给 AI 直接高权限”变成产品优势：

- 不乱发
- 不乱执行
- 不乱写记忆
- 不替老板乱承诺
- 每个高风险动作都有候选、确认、影响预览和回执

未来飞书、微信、CRM、ERP、财务软件、Frappe、Baserow、OCR、IM、Dendrite、OpenHands、Codex 等有开放接口的软件，都可以真实地成为 Provider、Surface 或外部能力来源。

即便是 Siri、Copilot、Gemini 这类系统级 AI，在哲学立场上 Truzhen 也不视其为对手，而视其为潜在的能力来源。它们目前并不开放、不可控，这句话不代表近期有具体集成计划，只表明 Truzhen 的定位是治理层，而不是权限竞争者。

Truzhen 的位置是连接层和治理层：

```text
OS / App / SaaS / Provider / 云服务
        -> 能力来源
Truzhen Gateway + Base Gate + Receipt
        -> 主权事务链
Owner
        -> 判断、授权、责任承担
```

## 6. 人、事、判断，而不是消息、文档、权限

协作软件的核心对象通常是人、组织、消息、文档和审批。Truzhen 的核心对象是人、事、判断、候选和回执。

| 维度 | 飞书类协作软件 | Truzhen |
| --- | --- | --- |
| 核心对象 | 人、组织、消息、文档 | 人、事、判断、候选、回执 |
| 工作方式 | 人在工具里协作 | AI 生成候选，人做主权裁定 |
| 自动化 | 审批、低代码、集成 | 能力执行 + 主权门 + 回执 |
| 记忆 | 文档和聊天记录 | 正式记忆 + 证据 + 综合页 |
| 事务 | 分散在群、表、文档 | 事务对象统一承载 |
| 风险控制 | 权限和审批 | FutureShadow + Base Gate + Receipt |
| 适用客户 | 中大型组织协同 | 老板亲自管事务的小团队 |
| 商业资产 | 应用生态 | 场景包 / 能力包 / 角色包 / 经验蒸馏 |

Truzhen 不是替代飞书、微信、CRM、ERP 或财务软件。Truzhen 是连接它们的主权事务层。

## 7. Pack 是人类经验给 AI 装上的边界

场景 Pack、能力 Pack、角色 Pack 的根本意义，不是给 AI 更多自由，而是把人类高手的经验、流程、红线和责任分寸变成可安装、可复用、可审计的边界。

三类 Pack 固定为：

- Domain Work Pack / Scene Pack：在什么业务情境下，按什么流程组织对象、角色和能力。
- Capability Pack：能做什么，风险是什么，需要什么 Gate、Provider 和 Receipt。
- Role Pack：AI 以什么行业角色、口吻、方法和权限边界提出候选。

不得发明第四种 Pack。

一个合格的场景 Pack 不是 prompt 模板，也不是“让最强模型一句话做事”的捷径。它必须回答：

- 什么对象是这门生意里的事务对象。
- 哪些动作只是候选，哪些动作必须正式化。
- 谁只是 Proposer，谁可以确认。
- 哪些风险必须回 Owner。
- 哪些 Provider 只是能力来源，不能当真相源。
- 哪些证据和回执必须留下。
- 哪些行业红线不能由 AI 自行判断。

所以 Pack 是行业高手把主权工作流提供给系统。它限制 AI 的自由发挥，让 AI 在行业经验和主权闸门中出力。

Pack 的边界不是在白纸上设计出来的，而是从服务真实客户的过程中长出来的。判断一个 Pack 该覆盖多大范围、该分几个工作模式、红线该划在哪里，应由真实场景证据决定，而不是由架构完整性决定。抽象层级高不是成就，是风险；唯一的成功标准是真实客户用起来了。

### 7.1 Pack 中的知识资产：行业高手卖的是领域语义

行业高手的经验里，最难被通用模型替代的不是流程，而是**这个行业的词和概念到底什么意思**：外行容易误解的术语、团队内部默认不解释的口径、概念之间的从属关系、判断和计算的约束与例外。这些「领域语义与隐性业务知识」（俗称"暗知识"）是 Pack 的核心知识资产，不是可有可无的附件。

大厂给一个企业中央建一套语义层；Truzhen 的路线相反——上万个细分行业，每个行业的语义由最懂它的行业作者装进他的 Pack，随 Pack 被购买、安装、治理和迭代。**Pack 生态就是分布式的语义层**，这是行业作者模式的又一个价值点。

承载不新增机制：语义是三类 Pack 都携带的横向内容维度（场景 Pack 是行业语义主载体，角色 Pack 携带专业判断边界，能力 Pack 携带动作含义与风险）；术语和口径以 `glossary` 等知识 kind 进入 Pack 知识域，AI 生成候选时经受控上下文裁剪消费，不得临场猜口径。没有来源和核验状态的"语义"，与没有证据的红线一样，不能进 Pack 冒充行业事实。

## 8. 红黄绿风险应按责任后果划分

风险分级不应按技术难度划，而应按“出错以后谁兜得住”来划。

- 绿：资料整理、摘要、草稿、内部建议、只读查询。出错可修，Owner 兜得住。
- 黄：普通业务流程、候选提交、Pack 声明、前端交互、低风险自动化。需要确认影响和回执。
- 橙：contracts、Gateway、Provider 接入、License / Entitlement、支付状态、跨仓边界。必须先做影响评估。
- 红：Base Gate、Receipt、真实执行、真实发送、真实付款、数据删除、权限放宽、生产部署。只能先提方案或红色计划，不能直接施工。

不同行业的红线不同。环保执法里一份文书错了可能是法律责任；装修报价错了可能是赔付责任；设计稿错了可能只是返工。统一系统不能替行业作者假装懂所有红线。

因此，Truzhen 的行业适配不应靠一个通用大模型拍脑袋，而应靠行业作者在 Pack 中声明该行业“什么错兜不住”。Base 负责执行硬边界，Owner 负责最终裁定。

## 9. 把 Owner 的秤装到系统里

AI 没有“做到 80 分够用”的内生分寸。它可以继续查、继续推演、继续改稿，因为消耗的时间、Token、算力和业务机会不是它自己的。

Truzhen 不指望 AI 学会心疼成本。Truzhen 要把 Owner 的分寸感做成系统参数：

- 每个任务有预算、轮次和停止条件。
- 高风险候选必须展示影响预览和替代方案。
- 低价值推演要能熔断。
- 默认追求可用闭环，而不是无限优化。
- 任务队列饱和时优先处理已有任务；低负载时再由 Goal Review 生成长期候选。

AI 没有秤，就把老板的秤装到系统上。老板的秤不是一段 prompt，而是预算、队列、门禁、回执和可回放的事实链。

这杆秤也要称量协调复杂度。多个 agent 互相协调的开销，常常大于它带来的收益；默认应是一个持续上下文的工作主体按需调用职能，而不是多个各自独立、需要互相协调的 agent。是否拆成独立 agent，应由“这些工作是否真的能并行、真的相互独立”决定，而不是由“看起来更像一个智能团队”决定。

## 10. 六仓如何承载这套哲学

Truzhen 的哲学不是一段宣传语，而是六仓身份边界。

| 仓库 / 目录 | 哲学职责 |
| --- | --- |
| `truzhenos` | 主权、闸门、回执、网关、运行时和装载器。它是权力中心，负责 Owner / Base / Gateway / Receipt 的真实实现。 |
| `truzhen-contracts` | 跨仓数据形状的权威源。它只声明 Pack / Candidate / Receipt / Surface / ReadModel schema 和 SDK 类型，不实现运行时。 |
| `truzhen-packs` | 行业高手经验资产仓。它声明场景、角色、能力、知识和安装脚本，不持 Base 主权，不实现 Provider。 |
| `truzhen-software` | 本机外部 provider / sidecar / runtime registry。它登记资源事实，不代表产品 ready，不拥有正式业务事实。 |
| `truzhen-cloud` | 官方云服务、云网页、市场、支付、License / Entitlement、Pack 分发和发布治理真相源。它不持本地主权。 |
| `truzhen-client-web-desktop` | Web/Desktop client layer。它展示 ReadModel、Candidate、GatedAction、Receipt，不自铸正式 ref，不直接执行或发送。 |

六仓共同形成一个原则：

```text
contracts 定形状
packs 定行业边界
software 登记资源
cloud 持云端事实
client 展示和确认
truzhenos 执行主权链
```

任何新功能都必须先问“事实归谁”。前端、Pack、Provider、contracts 默认不是真相源。它们只能展示、声明、提供能力、定义形状或生成候选。

依赖方向必须单向、不可逆，避免六仓重新焊死：

| 方向 | 规则 |
| --- | --- |
| `truzhen-contracts` | 不依赖任何实现仓，只提供跨仓形状和 schema。 |
| `truzhenos` | 实现 contracts；内部实现对其余五仓不可见；通过 `/v3`、Gateway、ReadModel、Candidate、Receipt 和受控文件加载协作。 |
| `truzhen-packs` | 面向 contracts 声明 Pack，不 import `truzhenos` 内部，不直连 provider，不依赖 cloud / client 内部实现。 |
| `truzhen-software` | 登记本机 provider / sidecar / runtime 资源，不被 packs 或 client 直接 import；产品 ready 由 `truzhenos` 裁剪后给出。 |
| `truzhen-cloud` | 可消费或实现 contracts 定义的云端形状，不 import `truzhenos` 内部，不伪造本地主权对象。 |
| `truzhen-client-web-desktop` | 只消费 contracts schema 和 `truzhenos` `/v3` API，不 import 基座 Go 内部、packs 内部、software raw provider 或 cloud admin 内部。 |

跨仓协作一律通过 contracts 定义的接口、已发布的 API、ReadModel、Candidate、GatedAction、Receipt 或明确的文件契约完成，不通过内部实现互相牵连。

## 11. Truzhen 的目标客户与商业机会

Truzhen 第一阶段最适合 3-50 人的小团队，尤其是老板亲自管事务、承诺、客户、项目、售后和外部执行的团队。

这类客户不是缺一个更会聊天的 AI。他们缺的是：

- 事情分散在微信、群、文档、表格、CRM、ERP、财务软件里。
- 老板必须为每个承诺和执行结果承担责任。
- 员工和工具都能产生动作，但责任最后回到老板身上。
- 行业经验很值钱，但难以复制给新员工或新团队。
- 小团队没有预算买大型定制系统，也不想让不可控 Agent 拿高权限。

Truzhen 的机会在边缘场景，但要把边缘做深。通过社区引导、行业作者和分账机制，把上万个细分行业的高手经验蒸馏成可安装、可复用、受主权闸门约束的事务资产。

理想形态是：

```text
少量核心开发人员维护基座和契约
大量行业作者维护细分行业 Pack
大量小团队用户安装 Pack 并沉淀真实回执与复盘
```

## 12. 面向用户的叙事

Truzhen 对用户不应主卖“AI 多自主”，而应主卖“责任清晰的效率”。

更合适的表达是：

- AI 把一切准备好，你来拍板。
- 每个高风险动作都先成为候选。
- 每次确认都有影响预览和回执。
- 每一笔承诺、发送、执行和写入都有证据可查。
- AI 不替你乱承诺、不乱发、不乱删、不乱写正式记忆。

对小团队老板来说，这比“全自动 Agent”更有价值。因为他真正焦虑的不是 AI 少做了几步，而是“手下人和工具干出的事最后都要我担”。

Truzhen 尊重这个焦虑，并把它做成系统结构。

## 13. 面向行业的叙事

行业正在从“自主 Agent 概念狂欢”回到“边界内执行、审计可见、人类拍板”。这不是倒退，而是企业级落地的常识回归。

Truzhen 应主动和无边界自主 Agent 划清界限：

```text
我们从第一天就不做无人拍板的 Agent。
因为你的生意，责任在你，拍板就该在你。
```

这不是保守，而是把商业世界最硬的责任事实放在架构第一层。

## 14. 非目标

Truzhen 不做以下事情：

- 不做无人拍板的自主 Agent。
- 不把 AI 包装成能承担责任的主体。
- 不和 OS 抢系统级权限。
- 不替代飞书、微信、CRM、ERP、财务软件或行业 SaaS。
- 不把 Pack 写成绕过基座的独立 App。
- 不把 Provider readiness 写成执行授权。
- 不把 cloud Entitlement 写成本地 Base Gate 裁定。
- 不把 client UI 写成真相源。
- 不把具体行业功能、具体设备的专用逻辑写进 `truzhenos`。`truzhenos` 只承载四类通用机制：主权内核、契约接口、装载运行时、通用基础设施。具体功能应长成 Pack 或 Provider；新接入面应复用设备无关接口，而不是在基座里长出专用分支。
- 不允许 AI 未经 Owner 确认就使自己起草的治理文件、风险分级或 Pack manifest 生效。规则的产生可以由 AI 辅助，规则的生效永远只能由人拍板。
- 不把 Receipt 写成 Memory。
- 不把 Candidate 写成 Formal。
- 不把 mock-only、demo-only、readmodel-only、candidate-only 写成已验收或已发布。

## 15. 最小闭环

Truzhen 的最小可交付闭环不是“AI 自动完成一切”，而是：

1. 用户的一件事被组织成 Transaction / Business Object。
2. AI、Pack、角色或模块生成候选。
3. 系统展示原因、风险、后果和可选方案。
4. Owner 当场确认，或使用预先设定的授权策略做确认、拒绝、修改或延后。
5. Base Gate 按策略裁定。
6. Gateway 受控调用模型、记忆、沟通或执行能力。
7. Receipt / Evidence 记录全过程。
8. 记忆只以候选进入，经 Owner 确认后才变正式记忆。
9. 前端只展示投影、候选、门控和回执。
10. Pack 作者通过真实行业经验持续改进场景边界。

这个闭环一旦成立，Truzhen 就不是“一个 AI 聊天工具”，而是一个可以承载真实事务责任链的操作层。

## 16. 终局形态

Truzhen 的终局形态可以概括为：

```text
本地优先 AI 事务操作层
+ 小团队老板工作台
+ 人类经验蒸馏与售卖平台
```

它处理的是小服务团队的客户、项目、售后、承诺、沟通草稿、执行授权和回执复盘。

AI 不是主人。OS 不是敌人。Provider 不是真相源。Pack 不是执行器。前端不是裁判。云不是本地主权。

Owner 是判断和责任的锚点。Base 是主权门。Gateway 是受控能力通道。Receipt 是证据。Pack 是行业经验给 AI 装上的边界。AI 是把判断前的一切准备好的工具智能。

这就是 Truzhen 的底层逻辑。
